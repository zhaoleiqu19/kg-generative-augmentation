# 对齐主张(Alignment Thesis)

> 状态:草稿,但思路已与作者敲定(2026-06)。纳入标准的具体取值标 "待定" 的部分,开始宽调研前再锁。

## 核心主张

诊断驱动的生成式增强流水线 = **诊断** + **生成** + 中间的**桥**(把诊断信号转成生成规格)。

> **若诊断输出的失败表示 = 生成器消费的控制接口(conditioning representation),则桥塌缩为轻量层。**

因此重点放在**诊断**和**生成**两端,并**同时**思考、找两端的**共享表示**;桥不是单独攻关的对象,而是这条共享表示轴的副产物。

## 为什么桥难 = 表示不对齐

桥之所以又重又易错(旧 G4),根因是诊断常输出"图像级聚类/属性标签",而生成器要"框/区域/layout 条件"——**两种语言**。让诊断直接用生成器的语言描述失败,桥近乎平凡。

## "对齐"的操作定义 + 配对

诊断输出表示 == 生成器控制接口。由此得配对地图(详见 [representation-map.md](representation-map.md)):

- **框/实例 layout** 生成 ↔ **框级/尺寸/遮挡/定位/类混淆** 诊断
- **区域 inpaint** 生成 ↔ **"哪个区域缺/漏检"** 诊断
- **自由文本/属性 prompt** 生成 ↔ **自然语言属性切片** 诊断
- **实例掩码** 生成 ↔ **像素/实例级** 诊断

## 定位用的轴(positioning axes)

判断一篇相关工作"和我们像不像"、差在哪,用这几条轴:
1. **任务**:检测 / 分类 / 分割(检测为主,分类仅旁证)
2. **诊断驱动**:是否由实测失败牵引生成(vs 盲增强 / 通用不确定性)
3. **闭环 vs 一次性**:诊断→生成→重训→**重新诊断** 闭合,还是静态一次性造数据
4. **对齐**:诊断输出表示与生成控制接口是否同一种语言
5. **表示标签**:框 / 区域 / 属性文本 / 实例掩码(宽调研时每篇都打)

## 纳入标准(宽调研用)

- 任务 = 检测优先;分类/分割仅作旁证。
- 时间窗:**待定**(倾向 2022+,但**按"是否有验证/代码/被引"加权,而非纯按时间**——新 ≠ 已验证)。
- 必须有**方法**(排除纯 benchmark / 纯数据集)。
- 每篇打一个**表示标签**(见上)。
- 经典/奠基论文(如检测误差分类法)**不作 SOTA 基线,但作"表示/分类法来源"保留**。

## 两个不能掉进的坑

1. **桥不会真归零**:就算表示对齐,仍有"哪些切片优先、各生成多少、怎么从一句话采样出具体框/场景"的**决策/采样策略**。桥从"重翻译"变"轻策略",但不消失。
2. **对齐是约束,不是唯一标准**:别为了好对齐选弱诊断/弱生成。终点是闭环跑出真增益;每端**自身要够强**(诊断找到真实可行动的失败;生成产出框正确、在分布内的数据)。对齐当加分项/平局决胜。

> 补:闭环靠**重新诊断**闭合,所以诊断输出还得是**可重复测量**的,才能判断生成有没有把洞填上。

## 与旧 GAPS 的关系

本主张是旧 G1(检测闭环)+ G2(检测粒度诊断)+ G4(切片→spec 桥)的**统一更锐表述**:贡献面 = **检测诊断与可控检测生成之间的对齐接口**。新 GAPS 见 [alignment-gaps.md](alignment-gaps.md)。
