## 一、目标与范围(来自设计规格)

构建一条**诊断驱动的生成式数据增强流水线**:给定一个(视觉模型, 任务, 数据集),自动**诊断**数据哪里弱、**生成**针对性增强数据、**过滤 + 重训**、并**测量**变化。先在一个简单公开检测任务上以"垂直切片"证明,但把模块接口设计成任务与各阶段方法日后可换。电梯 CCTV 电动车检测仍是最终目标,而非首次演示。

这**不是对此前工作的转向**:`20-loop/` 的四阶段(诊断 → spec → 合成 → 闭环)正是这条流水线;`90-papers/` 综述是"别人每步怎么做"的借鉴库。变的只是澄清:电梯任务成为*首个目标应用*,而"**哪些(视觉模型, 任务)对本方法友好**"成为一个显式的、检测优先的综述轴。

**已定方法:** 方案 A"垂直切片" + 方案 B"模块接口边界"——为一个 demo 任务搭一个最小的端到端运行,但从第一天起就定义干净、可换的模块契约。

### 本轮交付物
1. 综述收尾:一张**检测优先的通用性地图** + 一个选定的**公开 demo 任务**(模型 + 任务 + 数据集) + 四阶段各一个选定的现成组件。
2. 一条**最小端到端流水线**(四模块由契约连接)。
3. 一个**结果**:在 demo 任务上指标是否改善,以及为什么。

### 成功标准(完成定义 DoD)
- 流水线端到端跑完一整轮(诊断 → spec → 生成 → 过滤 → 重训 → 评估),并产出一份对比报告。
- 报告 demo 任务上的**基线 vs 增强**指标。**升 / 平 / 降都算有效**——成功 = 闭环闭合且结果可解释,而非必须改善。
- 模块边界成立:换任务或换某阶段方法,不需要改其它模块。

### 不在范围(YAGNI)
- 不一次支持所有视觉任务(检测优先;其它 = 仅相邻证据)。
- 不发明新诊断/生成算法——复用已发表、已验证的。
- 首版**不用电梯数据**(没有);不做大规模多任务实证(那是后续通用性验证)。
- 无 GUI/服务;一个 CLI 运行足够。

## 二、模块架构与数据契约

四模块 = 四个 loop 阶段,每个单一职责,只通过数据契约通信;内部实现可换。

```
            真实(模型, 带标注数据)
                     │
   1. 诊断 Diagnose   diagnose(model, dataset) -> list[Slice]
                     │  list[Slice]
   2. 规格 Spec       make_spec(slices, budget) -> GenSpec
                     │  GenSpec
   3. 生成 Generate   generate(spec) -> SyntheticDataset
                     │  SyntheticDataset
   4. 闭环 CloseLoop  close_loop(model, real, synth) -> Report
                     │  (过滤 → 累积真实+合成 → 重训 → 评估)
                     ▼
              Report(基线 vs 增强 + 决策)

   编排器 Orchestrator:跑 1→2→3→4,管理轮次 + 停止准则。
```

### 数据契约(唯一的耦合点)
| 类型 | 关键字段 | 生产者 → 消费者 |
|---|---|---|
| `Slice` | `description`(人可读,如"small + occluded")、`members`(样本 id 列表)、`metric`(该切片上的 AP/acc)、`severity` | 诊断 → 规格 |
| `GenSpec` | `items`:每个 = {目标切片, 生成条件(自由字段:prompt/layout/region), 数量} | 规格 → 生成 |
| `SyntheticDataset` | `images` + `annotations`(与真实集同格式,如 COCO json) | 生成 → 闭环 |
| `Report` | `baseline_metric`、`augmented_metric`、`per_slice_delta`、`iterate?` | 闭环 → 人/编排器 |

### 接口原则(B 边界)
- 每个模块是只遵守上述契约的可替换实现(如 `Diagnose` = HiBug2 实现 或 GH-ESD 实现)。
- **换任务**只动数据加载 + 评估指标(检测 = mAP / 尺寸相关 AP;分类 = acc);模块代码不变。
- **换方法**只改那一个模块的实现;契约不变——这正是垂直切片仍能验证边界的原因。
- `Slice` 用双表示:**样本 id 列表 + 人可读描述**(既定位样本*又*喂 prompt)。
- `GenSpec` 的生成条件是**自由字段**(prompt/layout/region 都能放),首版不用刚性 schema。

### 首版组件(综述给出的最终选择,**注:T3/T4 后部分已临时改动,见第五节**)
- **诊断:** HiBug2 或 GH-ESD(检测粒度切片)。
- **规格:** 简单规则(切片描述 → prompt/layout 条件;数量按 TADA 式放在最差切片上)。
- **生成:** ODGEN *或* DreamBooth + X-Paste(随 demo 任务与 Flux 能力探针定)。
- **闭环:** CLIP / 检测器自洽过滤 + 累积真实+合成 + 重训 + 评估。

## 三、验证、测试、错误处理、风险(来自规格)

### 测试(按契约分层)
- **单元:** 每模块用伪造契约测试(喂构造的 `Slice`/`GenSpec`,断言输出格式 + 边界);无真实模型/生成。
- **集成冒烟:** 极小数据(~几十张)跑一遍四模块;断言"端到端不报错、产出 `Report`",而非增益——CI 级门。
- **边界检查:** 把诊断换成 dummy 实现仍能端到端跑 → 证明解耦。

### 验证"有用性"(科学严谨)
- 固定随机种子;同一基线 + 同一评估集。
- 比较基线 vs 增强;报告 `per_slice_delta` + 总体。
- 反自欺:**评估集不含合成图像**;关注目标切片的 delta,而非被稀释的总体均值。

### 错误处理(流水线鲁棒性)
- 生成差 → 闭环过滤(CLIP / 检测器自洽)拦截;若存活太少,**带警告跳过该轮**而非污染训练集。
- 重训发散/下降 → `Report` 记"down"、`iterate?=false`;不强制多跑。
- 累积默认:**真实 + 合成累积,绝不替换真实**(避免坍缩;Gerstgrasser et al. 2024)。

### 风险缓解
| 风险 | 缓解 |
|---|---|
| 没有电梯数据 | 垂直切片用公开集;电梯留在通用性地图作目标行,待数据 |
| Flux 能力未知 | 生成模块可换;回退到 ODGEN / SD + X-Paste;先跑 Flux 能力探针 |
| 算力不足 | 小数据集 + 预训练微调 + 单轮起步 |
| demo 任务选错(本就难改善) | 选择清单要求"有人报告过增益",借别人的结论 |

## 五、T3/T4 决策详情 + 课程修正(来自 decisions/ 与 SDD 账本)

### T3 决策 — demo 任务选择(临时)
- **选定 demo 任务:Pascal VOC,低数据 / few-shot 检测。** 数据集 VOC 2007+2012(20 类),用**低数据子集**(k-shot 或一小部分 trainval),既触发 gen-aug 最有用的区间又让单 GPU 够用。
- **为何:** 最易搭起、单 GPU 快迭代、VOC 上有已发表 gen-aug 增益(可控扩散+CLIP +2.9% 全量;~+15–18% few-shot)、所有候选生成器都在 VOC 类检测上演示过。
- **评估集:** 标准 VOC test/val,**不含合成图像**。主指标 mAP@0.5 总体 + **逐类 AP**(逐切片信号)。
- **基线检测器:** Ultralytics YOLO(如 YOLOv8n/s),**预训练权重微调**于低数据 VOC 子集(单 GPU 友好、快、API 文档好;不从零训)。
- **逐模块选择:** 诊断 = **轻量切片器**(按 AP 排类 × 目标尺寸带:小/中/大),可复现、无重外部库(借 Domino/HiBug2 思路;升级路径 = 经同一 `Diagnoser` 接口换 HiBug2/GH-ESD)。规格 = `RuleSpecifier`(按严重度分预算,封顶)。生成 = 见 T4。闭环过滤 = **类别校准 CLIP 分数**。
- **预算:** 单轮,只为最差 1–2 个切片生成适量(每切片 ~50–200 张)。

### T4 决策 — Flux 能力探针(临时,GPU1 实测)
- **设置:** 模型 `/data1/.../FLUX.2-klein-4B`,conda `flux2`(py3.11,torch2.6 cu124),脚本 `generate.py`/`batch_generate.py` = **纯文生图**(无 ControlNet/layout、无 img2img、无 DreamBooth/LoRA)。模型加载 ~5s,4 步 ~8–45s/张。
- **发现 A — 平背景孤立目标:** 结果干净、居中、近纯白无缝背景。判定**极佳**(易分割成掩码 → 理想 copy-paste 前景)。
- **发现 B — 经 prompt 的粗略空间控制:** 目标确实被渲染在请求的角落、其余场景空。判定**粗略位置控制可行**(区域/角落级),但**不是像素精确 bbox 条件**(能引导*在哪*,但要紧框仍须*检测*生成的目标)。
- **个性化(DreamBooth/LoRA):** 本地脚本没有,**未探**;VOC demo(通用类)不需要;电梯阶段对特定实例才需加 LoRA/DreamBooth。
- **决策 — Generate 模块 = copy-paste 路线(原样用本地 Flux):** ① 前景:本地 Flux 在平背景生成弱类的干净孤立实例;② 掩码:平背景 → 简单阈值分割抠出;③ 合成:贴到真实 VOC 背景的选定位置 → **边界框精确已知**(我们放的),绕开 Flux 缺像素精确 layout;④ 过滤:类别校准 CLIP 分数。**理由:** copy-paste 免费给精确紧框,只需现有本地 Flux T2I,无新重库、无吃显存的微调。
