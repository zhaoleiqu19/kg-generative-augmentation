# 诊断驱动的生成式数据增强 —— 思路与文献

> 组会分享 · 2026-06 · 标 ★ 为各部分代表作

## 0. 一句话课题 + 核心主张

- **课题**:给检测模型做"诊断驱动"的生成式数据增强。**锚点任务 = 电梯 CCTV 检测电瓶车/电动车**(近景、遮挡、暗光、拥挤)。
- **核心主张(对齐 Alignment)**:一条增强流水线 = **诊断** + **生成** + 中间的**桥**(把诊断信号变成生成规格)。
  > 如果**诊断输出的"失败描述" == 生成器消费的"控制接口"**(同一种语言),那中间这座桥就塌缩成很轻的一层。
- 所以我们把**诊断和生成放在一起看**,找两端的**共享表示**,而不是单独硬啃那座桥。

---

## 1. 整体 Loop —— 这个方向大家在做什么

**这个"环"是什么(四步):**

```
① 诊断 —— 模型在哪类样本上栽了?(找弱点)
② 生成 —— 照弱点造对应新图(+ 正确的框)
③ 过滤 —— 留好的、扔坏的
④ 重训 —— 训回去 →(回 ①)再诊断,确认补上了
```

关键不在某一步炫技,而在 **① 的输出能不能直接当 ② 的输入**。

**两种"最接近我们"要分开看**:① 闭环完整度(真把环转完整);② 对齐度(把诊断和生成焊在一起)。没有一篇两者都占满。

### 1.1 整环近邻(诊断/反馈信号 → 生成)

- ★ **SDRS — Synthetic Designed Experiments for Diagnosing Vision Model Failure**(Sarkar, 2026,投 CVPR SynData4CV)— 把生成器当"实验仪器",用 **DoE / 部分因子设计**主动控制光照、视角、遮挡等因子,再对下游模型 loss 做 **ANOVA**,输出**因子敏感度画像**;把失败分成 **Type I 覆盖缺口** 与 **Type II 捷径(spurious)缺口**,并对症开"targeted 合成数据"处方。**非常贴"先诊断再生成"**,但依赖**可控合成因子**,不是从真实检测错误里归纳 grounded slice。
  🔗 https://arxiv.org/abs/2605.00832
- ★ **DetDiffusion**(CVPR 2024)— 把感知模型(检测/分割)的信号**焊进**扩散生成,生成天然偏检测器所需;**对齐度最高**,但信号通用、且一次性不回头。
  🔗 https://arxiv.org/abs/2403.13304
- ★ **LTDA-Drive**(2025)— 生成稀有类→插入→**LLM 过滤**→重训,长尾 3D 检测,KITTI 稀有类 +34.75%;**闭环完整**,但由"盲长尾"牵引。
  🔗 https://arxiv.org/abs/2505.18198
- **GALOT**(2024)— 明确"主动学习准则→生成"真闭环;但分类、纯合成、无框。
  🔗 https://arxiv.org/abs/2412.16227
- **SafeFix**(2025)— 诊断→针对性生成→VLM 过滤→重训,已发表的"修模型"闭环模板;以分类为中心。
  🔗 https://arxiv.org/abs/2508.08701
- **AUTHENTICATION**(2025)— 找感知失败模式→对抗引导 inpaint 生成"骗过检测器"的场景;诊断=对抗造假,非实测。
  🔗 https://arxiv.org/abs/2504.17179
- **Chegini**(2023)— 诊断→CLIP 辅助扩散生成补难子群;分类。
  🔗 https://arxiv.org/abs/2312.05464
- **Improving-OD with XAI**(2024)— XAI 显著图→人手改 3D mesh→重渲染;反馈驱动,但渲染图 + 人在环。
  🔗 https://arxiv.org/abs/2412.01477

### 1.2 位置近邻(只解决"往哪儿放")

- **Scene-Aware**(2025)— 学概率位置模型 + inpainting,+1.4 mAP(比随机放好 2.8×);位置来自场景先验,非诊断。
  🔗 https://arxiv.org/abs/2504.17076
- **DIAG**(CBMI 2024)— 人手框区域,免训练在分布内造缺陷,+18~28 AP;= "切片→规格"标记的人工版。
  🔗 https://arxiv.org/abs/2407.03961

### 1.3 纯生成引擎(只管造得好,不碰诊断)

- **DiffusionEngine**(2023)— 单阶段产"图+框",+3.1 mAP(COCO);盲扩规模。
  🔗 https://arxiv.org/abs/2309.03893
- **ODGEN**(NeurIPS 2024)— object-wise 条件,能造多类+遮挡;域内 +25.3 mAP。
  🔗 https://arxiv.org/abs/2405.15199
- **ReCon**(NeurIPS 2025)、**AeroGen**(遥感)、**X-Paste**(copy-paste)。
  🔗 https://arxiv.org/abs/2510.15783 · https://arxiv.org/abs/2411.15497 · https://arxiv.org/abs/2212.03863

### 1.4 泼冷水 / 综述(校准预期)

- **Big-Data-Myth**(2023)— 简单任务上纯合成接近真实(AP 差 0.09~0.12);合成更多是"替代",别高估。
  🔗 https://arxiv.org/abs/2306.09762
- 综述:**UniDiffDA**(2026,仅分类) 🔗 https://arxiv.org/abs/2603.08364 · **Alimisis Review**(2024) 🔗 https://arxiv.org/abs/2407.04103

> **整体层结论**:环的每一段都有人做过,**但没有一篇把"实测的、grounded(落到具体框)的检测诊断"接到"可控、能处理遮挡、真实图"的生成器 + 明确验证器闭环**。这道缝就是我们的位置。

## 2. 诊断端 —— 模型差在哪("切片")

**什么是切片(error slice):** 一组**有共同、可描述特征、且模型在上面成片栽**的样本。例:"被电梯门半遮挡的电瓶车""暗光+黑色车身"——不是错一两个,是这一类整体差。切片就是"诊断的输出",越能**落到具体物体/框**(grounded)越好。

**筛选文献(框级):**

- ★ **GH-ESD**(ECCV 2026)— **实例级 grounded** 失败切片发现(LLM 造假设→VLM 验证):把失败落到"被门遮挡的电瓶车"这种具体物体+空间关系;**最接近能直接当生成规格的形态**。
  🔗 https://arxiv.org/abs/2512.24592
- ★ **TIDE**(ECCV 2020)— 检测误差的标准"词汇表":分类/定位/两者/重复/背景/漏检 六类,并隔离每类对 mAP 的贡献;是 dataset 级聚合。
  🔗 https://arxiv.org/abs/2008.08115
- **HiBug2**(2025)— tag-then-slice,生成属性→枚举切片,且能**预测验证集之外的切片**(对锚点的未见失败有用);表示是属性文本。
  🔗 https://arxiv.org/abs/2501.16751
- **FN-Mechanisms**(RA-L 2022)— 打开检测器黑盒,命名导致漏检的 5 种内部机制;发现 benchmark≠部署。
  🔗 https://arxiv.org/abs/2203.07662
- **KGFP**(CVPR-W 2026)— 运行时预测何时漏掉安全攸关目标(检测器特征 vs 基础模型嵌入的角度失配);person recall 64.3%→84.5%;更像**触发器/验证器**。
  🔗 https://arxiv.org/abs/2603.25499
- *(分类参照)* **Domino**(ICLR 2022)— 跨模态嵌入发现系统性误差切片的代表作。
  🔗 https://arxiv.org/abs/2203.14960

> **诊断端小结**:失败的**词汇有了**(TIDE 的六类),但多是 **dataset 级聚合**;**grounding(落到具体框)偏少**,GH-ESD 最接近"切片→规格"。

## 3. 生成端 —— 按规格造图

**筛选文献(框/layout 控制接口):**

- ★ **InstanceDiffusion**(CVPR 2024)— 逐实例控制:每个物体单独给 **点/框/掩码 + 自由文本**;COCO 框/掩码大幅超前 SOTA。接口正是 grounded 诊断会吐出的语言。
  🔗 https://arxiv.org/abs/2402.03290
- ★ **OcclusionFormer**(ICML 2026)— 显式建 **Z-order 遮挡**(实例解耦+体渲染);补上多数 box 引擎缺的遮挡序,**直接打锚点**(被遮挡的电瓶车)。
  🔗 https://arxiv.org/abs/2605.21343
- ★ **GeoDiffusion**(ICLR 2024)— 文本+几何(框)控制做检测数据,**本地已可跑**,训练快 4×;box-only。
  🔗 https://arxiv.org/abs/2306.04607
- **MIGC++**(TPAMI 2024)— 多实例:框&掩码定位 + 文本&图像属性,divide-and-conquer 治"属性串味"(多相似实例时稳)。
  🔗 https://arxiv.org/abs/2407.02329
- **ODGEN**(NeurIPS 2024)— object-wise 文本+视觉,能造多类+遮挡;需逐域微调。
  🔗 https://arxiv.org/abs/2405.15199
- **DetDiffusion**(CVPR 2024)/ **DiffusionEngine**(2023)— 感知信号拌进生成 / 单阶段产图+框(盲扩)。
  🔗 https://arxiv.org/abs/2403.13304 · https://arxiv.org/abs/2309.03893

> **生成端小结**:**逐实例接口 + 遮挡序已成熟**(InstanceDiffusion / MIGC++ / OcclusionFormer),**本地 GeoDiffusion 可立即跑**。缺的不是引擎。

## 4. 诊断 ↔ 生成 的关系(对齐 + 缝在哪)

**两端的"语言"对照**:

| 生成器的控制接口 | 对齐的诊断输出 |
|---|---|
| 框 / layout(类别+bbox) | 框级 / 尺寸 / 遮挡 / 定位 / 类混淆 |
| 区域 inpaint(背景+mask) | 哪个区域漏检/易错 |
| 自由文本 / 属性 prompt | 自然语言属性切片 |
| 逐实例掩码 | 像素/实例级失败 |

**框级行已对账两端,结论:**

- **生成端已成熟**:能吃"逐实例 框/掩码 + 属性 + 遮挡序"。
- **诊断端词汇有、grounding 偏少**:GH-ESD 最接近,但还没有人把它直接变成生成规格。
- ★ **交叉状态 = 可交叉,也就是 gap**:两端语言相同,**但没有现成方法把"实测 grounded 检测诊断切片"自动翻译成"逐实例生成规格"去驱动 InstanceDiffusion / OcclusionFormer**。现有的要么自己造框(DiffusionEngine)、要么用通用/对抗信号(DetDiffusion / AUTHENTICATION)。

> **这条"诊断切片 → 逐实例生成规格"的轻桥,就是我们想做的点。**

## 5. 我们的初步判断 + 想听大家意见

**初步判断:**
1. 引擎和诊断都不缺,缺的是**把两端用同一种"框级/实例级"表示接起来的轻桥**。
2. 锚点(电梯)的硬点是**遮挡+近景+暗光**,正好 OcclusionFormer / InstanceDiffusion 这条线能承载。
3. 闭环还需要一个**验证器/触发器**(KGFP 这类)来判断"洞补上没有"。

**想请大家拍砖的开放问题:**
- 这个"切片→逐实例 spec"的桥,落地时该多"自动"?全自动 vs 半人工标注?
- 锚点选电梯电瓶车合适吗?有没有更好验证"诊断驱动 > 盲增强"的小场景?
- 验证器用 KGFP 这种"特征失配",还是直接用重训后的再诊断指标?
- 评测怎么设计才能证明增益来自"诊断对齐",而不是单纯多了数据?

## 6. 三篇关键文章给我们的定位

**DetDiffusion / GH-ESD / SDRS 放在一起看,会把我们的位置压得更清楚:**

- **DetDiffusion** 已经证明"检测器信号可以帮助生成更适合检测训练的数据"。它的诊断信号主要是实例级 **easy / hard** perception-aware attribute,能告诉生成器"这个对象对检测器难不难",但不显式解释 **why hard**。所以它会是强 baseline:如果我们只做 hard sample generation,就会和它冲突;我们的差异必须落在"把 hard 解释成可描述、可控制的失败原因"。
- **GH-ESD** 已经证明"真实检测/分割错误可以被整理成 grounded、instance-level、自然语言 failure slice"。它的输出大致是:一个 validated hypothesis(如"bicycle partially occluded by a person") + 每个实例/区域的 matching score + 统计验证结果。它强在诊断,但 repair 主要是 reweighting / GroupDRO,没有把 slice 翻译成 diffusion layout / mask / z-order / photometric spec 去生成新数据。
- **SDRS** 证明"合成数据最有价值的地方不是便宜,而是可控因子能做诊断实验"。它输出的是 factor-level sensitivity profile,例如遮挡是 Type I coverage gap,背景复杂度是 Type II shortcut gap,然后给出 targeted data prescription。它强在实验设计和因子归因,但前提是生成器已有明确可控 knobs,且实验主要是 dSprites / procedural segmentation,不是从真实部署错误中发现 grounded slice。

> **因此我们的更稳定位**:不是重新发明 DetDiffusion 的 detector-aware generation,也不是重新做 GH-ESD 的 slice discovery,而是研究 **"真实 grounded failure slice → 可控生成 spec → 合成数据修复 → slice-level 再诊断"** 这条桥。DetDiffusion 提供生成增强 baseline,GH-ESD 提供真实错误切片入口,SDRS 提供"哪些因子值得生成"的实验设计思想。
